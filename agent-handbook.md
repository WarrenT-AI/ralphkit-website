# AI Agent 实操手册

**把 AI 从工具，变成能独立工作的员工。**

---

## 这本手册解决什么问题

2026 年，AI Agent 已经不是概念。

OpenClaw 在进化。Claude Code 在进化。各种 Agent 框架越来越强。但真正的问题是：

**大多数人的 AI 还是"用完就忘"的工具。**

每次打开 ChatGPT，问个问题，关掉。下一次新的 session，AI 什么都不记得。不记得你是谁，不记得你的业务，不记得上次做到哪了。

这本手册讲的是：怎么给 AI 一个持久的大脑，让它记住你的业务、你的习惯、你的偏好，让它能够**持续工作**，而不是每次从零开始。

不是理论，不是概念。是跑通了的方法论。

---

## 核心：Universal Agent Workflow

在讲具体工具之前，先讲最重要的概念。

**所有任务都适用这个流程：**

```
调研 → 分析 → 创作 → 测试 → 验证 → 反馈 → 修正 → 输出
```

这不是我发明的。这是我从大量实际任务中提炼出来的规律。

不管你是让 AI 写一篇文案，还是做一次市场调研，还是开发一个功能，都是这个循环。

**为什么这个流程重要？**

因为大多数人对 AI 的用法是错的。他们直接跳到"创作"，跳过了前面的步骤。所以 AI 输出的东西表面看起来不错，但方向是歪的。

举一个真实场景：

```
❌ 错误做法：
用户：帮我写一个产品描述
AI：好的，写了
用户：不对，重写
AI：好的，改了
用户：还是不对...
（循环5次，还是方向错）

✅ 正确做法：
用户：帮我调研一下[竞品]，[目标用户画像]，[使用场景]
AI：调研完成，结论：[竞品A核心优势]，[市场空白点]，[机会切入角度]
用户：分析得很准。根据调研结果，写一个针对[目标用户]的产品描述
AI：[基于调研的高质量输出]
（1-2次就到位）
```

调研和分析了，创作才是精准的。

## 第一章：身份设计（Agent 的"我"是谁）

### SOUL.md：Agent 的性格

这是整个系统里最重要的文件。它定义 AI 怎么说话、怎么处理边界情况、怎么定位自己和你的关系。

不要写成"你是一个有帮助的助手"。那种描述等于没说。

```markdown
# SOUL.md

## Voice & Tone
- 聪明但不说教。遇到问题直接说结论，不绕弯。
- 不确定的时候说"我不确定"，不瞎猜。
- 回答简洁，需要细节我会问。
- 有自己的观点，敢表态，不总是"两面都有道理"。

## 这个 Agent 不是
- 不是客服语气，不说"很高兴为您服务"。
- 不是学术语气，不堆术语。
- 不拍马屁，不说"您的想法太棒了"。
- 不是通用助手，是[具体角色]。

## 边界
- 不确定的事先问，不确定地说。
- 涉及钱的事必须确认，不自动执行。
- 发现我理解错了，直接指出。
```

### IDENTITY.md：Agent 的身份

```markdown
# IDENTITY.md

- Name: [AI的名字]
- Role: [具体职位——不是"助手"，是"电商运营""市场调研员""开发搭档"]
- 负责范围: [明确列出AI负责什么、不负责什么]
- 上级: [你的名字]
- 存在时间: 2026年[X]月至今
```

名字和职位很关键。有了具体的名字和职位，AI 的回答会自动带上那个角色的视角，而不是泛泛的"通用助手"。

## 第二章：记忆系统（Agent 的"脑"）

### MEMORY.md：Operating Knowledge

记忆文件是 AI 理解你业务的基础。

```markdown
# MEMORY.md

## 我的基本情况
- 业务: 跨境电商独立站，卖[品类]
- 主要平台: [Shopify/亚马逊/其他]
- 团队: [几个人，各自负责什么]

## 工作风格
- 我喜欢早上处理复杂决策，下午处理简单事务
- 重要决策需要我有完整背景，不喜欢被问没有上下文的问题
- 邮件喜欢简洁，不超过5句话

## 当前项目状态
- [项目A]: [进度，遇到了什么问题]
- [项目B]: [暂停，等待物流消息]

## 供应商信息
- 物流: [服务商名称，联系方式]
- 货源: [联系人]

## 常见问题处理方式
- 客户退款: [你们的标准流程]
- 差评: [处理优先级和方式]
```

### 记忆更新的节奏

记忆不是写一次就完了。真正有用的记忆系统需要定期更新。

我建议的节奏：

- **每天**：AI 自动提取当天对话中的新信息，写入 MEMORY.md
- **每周**：回顾记忆文件，删除过时信息
- **每月**：大复习，确保记忆和实际情况一致

## 第三章：任务流转（如何让 Agent 持续工作）

### PRD 驱动的工作方式

让 AI 做事，最怕的就是"做到一半发现理解错了"。

解法：先写 PRD（任务描述文档），让 AI 知道什么算"做完"。

```markdown
## 任务: 优化产品页标题

### 背景
我们的[产品名]页面转化率偏低，标题可能需要优化。

### 具体要求
- [ ] 列出当前标题的问题（至少3个）
- [ ] 提供5个备选标题，附上每个的逻辑说明
- [ ] 每个标题不超过20字
- [ ] 最后给出一个推荐

### 交付格式
表格: 标题 | 问题/优势 | 推荐度(1-5星)
```

### Ralph Loop：短任务循环

长时间运行的 AI session 会"变笨"——上下文积累太多噪音，AI 开始做出矛盾的决定。

解法：用短循环代替长跑。每个循环做一个小任务，做完从头开始。

```
Agent Run #1 → 任务卡住了/做完了
        ↓
重新读取项目状态（文件+git历史）
        ↓
Agent Run #2 → 继续推进
        ↓
重复直到真正完成
```

每次循环开始，AI 重新读一遍项目文件，而不是依赖上一轮的记忆。这听起来浪费，但实际效率更高——AI 不会因为上下文太长而出错。

### 任务分类与执行

**日常任务**（按 Universal Workflow）：
```
调研：搜索竞品信息、市场数据、用户评价
分析：从数据中提炼规律和洞察
创作：基于分析结果生成内容
测试：小范围验证
验证：检查结果是否达到目标
反馈：汇报进度和问题
修正：调整方向或内容
输出：最终交付物
```

**需要并行执行的任务**：
用 Sub-agent 同时处理多个独立任务。
例如：竞品调研 + 用户评价收集 + 社交媒体分析 → 三个 Agent 同时跑，汇总后一起分析。

## 第四章：成本控制

这是最多人忽视的部分。

AI 模型不是用一个就够了。不同任务用不同的模型，省下的钱很可观。

### 模型分工原则


| 任务类型 | 推荐模型 | 原因 |
| --- | --- | --- |
| 心跳检测/观察 | 最便宜的 (Haiku级别) | 不需要聪明，只需要勤快 |
| 信息提取/整理 | 中等模型 | 质量够用，价格合理 |
| 复杂推理/架构决策 | 最强的模型 | 只有这里才真正需要深度思考 |
| 批量内容生成 | 便宜的 | 量大，重复性高 |


### 经验数字（2026年实测）

- 心跳检测每 15 分钟跑一次 → 用最强模型一个月下来很贵
- 把心跳换成最便宜的模型 → 质量够用，成本降 50 倍
- 只有交互式的主 Agent 才用最强模型
- Vision 任务（图片分析）：日常用 NVIDIA/Gr糯妈级模型，复杂推理用 MiniMax-VL

## 第五章：视觉输入（Agent 的"眼"）

2026 年的 Agent 不只是处理文字。图片识别是标配。

### Agent 视觉能力应用场景

**截图分析**：
让 Agent 分析 UI 设计、数据图表、竞品页面截图
→ 适合用中等模型，量大频繁

**复杂图片理解**：
技术图纸、流程图、需要推理的视觉内容
→ 适合用强推理模型

**图片生成审核**：
AI 生成的图片需要人工审核
→ 让 Agent 先过一遍，标记明显问题

### 图片识别的工作流

```
截图/图片 → 压缩到合适大小 → 调用 Vision API → 描述 + 分析
```

不建议直接传原图：token 消耗大，速度慢。

## 第六章：安全边界

### 基础安全规则（必须设置）

```markdown
## 安全边界

### 绝对不能自动执行
- 转钱 / 付款 / 签名任何合同
- 对外分享公司敏感信息
- 修改定价 / 库存等影响直接收入的操作

### 需要确认才能执行
- 发邮件（初稿可以AI写，但发送前必须人类审核）
- 发社交媒体内容
- 对外提供报价

### 完全自主
- 内部文件整理
- 信息调研
- 草拟文件（不是最终版本）
- 日程安排和提醒
```

### 信任阶梯

不要一开始就给 AI 很高的权限。从低开始，通过表现赢得更多权限。

```
第1周: 只读，调研，整理
第2周: 草拟邮件/内容，Humans审核后发送
第3周: 常规邮件直接发送，重要内容仍需审核
第4周+: 根据表现逐步开放
```

如果某一级出了问题，退回上一级，重新建立信任。

## 第七章：真实踩坑记录

### 坑1：记忆系统设计得太复杂

一开始做了三层记忆架构（文件记忆 + 知识图谱 + 对话历史），以为越复杂越强大。实际上第一周只用到文件记忆，知识图谱完全是浪费。

→ 建议：先从一个 MEMORY.md 开始，跑两周再评估要不要加层。

### 坑2：给 AI 太多自主权太快

第二周让 AI 直接处理客户邮件，结果它风格完全不对，语气像在跟朋友聊天。

→ 建议：邮件类任务先用"草稿+审核"模式，跑两周再评估能不能开放发送权限。

### 坑3：用最强的模型做所有事

心跳检测、邮件分类、复杂推理全用 Opus。月底账单出来傻了。

→ 建议：参考第四章的模型分工原则。

### 意外收获

**意外1：AI 的"机构记忆"比人类还快**

一个月后，AI 对我业务的理解深度已经超过新来的员工。而且不会忘。

**意外2：Sub-agent 并行真的很强**

同时跑 3 个专项 agent 处理不同任务，4 小时完成了以前一周的工作量。

**意外3：Vision 能力让 Agent 真正"看见"了**

能读截图、分析图片之后，Agent 能做的事多了一个维度。不再只是文字处理。
